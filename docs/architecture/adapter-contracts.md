# Adapter Contracts

CoMath integrates maintained tools through adapters. Adapters provide hints, candidates, evidence, and diagnostics; they never become proof authority by themselves.

## Common Adapter Envelope

Every adapter result must include:

- `adapter_id`
- `provider`
- `capabilities`
- request hash and response hash
- timestamp
- version or API endpoint when available
- license/terms note when content is external
- `proof_authority: "none"`
- artifact or evidence IDs for stored outputs
- blocker details when unavailable

Adapter failures are replayable blockers or degraded capability reports, not silent success.

## Theorem Search Adapters

Providers may include Loogle, LeanSearch, LeanSearchClient, Moogle, LeanExplore, LeanDojo traces, local mathlib indexes, and local external-repo indexes.

Allowed capabilities:

- constant and name search
- subexpression search
- natural-language theorem search
- proof-state search
- premise retrieval

Theorem search results may suggest imports, declarations, lemmas, or proof routes. They may not certify that a theorem is proved or that a statement is equivalent to the FormalSpecLock.

## Proof-Search Backend Adapters

Providers may include LeanRunner, Lean LSP/MCP adapters, LeanDojo, LeanCopilot, Aesop/tactic runners, Codex-compatible backends, and other configured proof-search systems.

Backend output must be an AgentCandidatePack or strict JSON equivalent with:

- locked statement hash
- candidate statement hash
- introduced assumptions
- introduced dependencies
- requested Lean checks
- hard vetoes and blockers
- `proof_authority: "none"`

Any Lean source from a backend must be rechecked by the service-owned LeanRunner and final replay gate.

## Agent Adapter OS Isolation

Agent adapter package launch metadata records whether an adapter currently runs behind `process_boundary_only` or has service-owned OS-isolation evidence. A release-readiness review may accept OS-enforced isolation only when the evidence is produced by a `comathd` service-owned collector and is bound to adapter id, backend, provider, probe id, stdout/stderr/transcript hashes, process isolation, filesystem scope, network isolation, no-new-privileges, and escape-prevention checks.

Provider-specific sandbox launch preflight is a separate service-owned manifest. It may record that a configured provider such as OCI, Nix, Firejail, Windows AppContainer, or macOS `sandbox-exec` is ready for a future service-owned execution probe, but it does not by itself record adapter execution, kernel/firewall isolation, or GA readiness. Preflight manifests must keep `shell=false`, `network_policy=disabled`, `command_override_allowed=false`, `caller_supplied_success_allowed=false`, `proof_authority="none"`, and `can_certify_ga=false`.

Provider host capability probe manifests record service-observed host capability facts for an adapter OS-isolation provider family before helper host validation. They may include platform applicability, provider-family tool presence, kernel/platform feature observations, sanitized diagnostics, and replayable blockers. The production route path uses a default service-owned host capability probe when no test callback is injected; that default may record the service-observed platform, provider support-table match, bundled provider-helper protocol asset presence/hash, and non-authority diagnostics. For `windows_appcontainer` on Win32 service hosts, the default probe may also record Windows AppContainer host facility diagnostics and `windows_checknetisolation` tool presence/hash metadata without exposing system paths. For `oci_container` on service-supported Linux, macOS, or Windows hosts, the default probe may record OCI container host facility diagnostics plus Docker/Podman CLI presence/hash metadata without executing those tools, inspecting daemons or sockets, recording versions, or exposing executable paths. For `nix_sandbox`, `firejail`, and `macos_sandbox_exec` on service-supported hosts, the default probe may record facility diagnostics plus `nix`, `nix-store`, `firejail`, or `sandbox-exec` executable-candidate presence/hash metadata without executing those tools, reading store/profile/sandbox-policy state, recording versions, or exposing PATH/executable paths. Route callers cannot submit platform strings, tool paths, kernel-feature booleans, or success-shaped payload metadata as capability evidence. Pi consumers for this route are host-confirmed thin clients and may forward only sanitized `platform` / `notes` diagnostics, not caller success-shaped capability fields. The manifest is host-capability diagnostics only: it is not provider-helper readiness, not helper self-test or runtime attestation, not executed OS isolation, not collected OS-enforcement evidence, not readiness-review evidence by itself, not mathematical proof authority, not real-Pi execution evidence, and not GA certification. It must keep `adapter_execution_isolation.os_enforced=false`, `proof_authority="none"`, `can_promote_claim=false`, and `can_certify_ga=false`.

Provider-runner contract manifests bind a ready sandbox-launch preflight to a fixed provider argv template and service-owned environment policy. They reject caller-supplied command, argv, env, runner hash, and success metadata. A host operator may configure absolute service-owned helper executables through provider-specific handles such as `COMATH_AGENT_ADAPTER_OSISO_OCI_HELPER`, `COMATH_AGENT_ADAPTER_OSISO_NIX_HELPER`, `COMATH_AGENT_ADAPTER_OSISO_FIREJAIL_HELPER`, `COMATH_AGENT_ADAPTER_OSISO_WINDOWS_APPCONTAINER_HELPER`, or `COMATH_AGENT_ADAPTER_OSISO_MACOS_SANDBOX_EXEC_HELPER`, with `COMATH_AGENT_ADAPTER_OSISO_PROVIDER_HELPER` as a host-only fallback. When no env helper is configured but the built package includes the bundled CoMath provider-helper protocol asset, the default resolver may bind the Node executable hash and bundled helper args-prefix hash/count as service-owned protocol material. The default resolver records only executable hashes, prefix hash/counts, and non-secret diagnostics, never helper paths or prefix argument values. Provider-runner manifests still do not record adapter execution, cannot satisfy the readiness gate, and must keep `adapter_execution_isolation.os_enforced=false`, `proof_authority="none"`, and `can_certify_ga=false` until a later service-owned execution probe writes canonical collected evidence.

Provider-helper host-validation manifests bind a ready provider-runner manifest back to its sandbox-launch artifact, selected provider, service-owned helper binary hash, supported host platform list, fixed argv template hash, non-inherited environment policy, and a matching service-owned provider host capability probe. Host validation fails closed before the host validator runs unless the referenced append-only host capability probe is observed and matches the current project id, adapter id, backend, provider, service-observed platform, artifact path/hash, and non-authority flags. The default host validator uses the service-observed host platform, not caller-supplied route platform strings, and fails closed when platform-specific helpers are configured on incompatible hosts. For env-configured helpers and bundled protocol assets, the default host validator also runs a fixed CoMath provider-helper self-test with service-owned argv/env; the self-test stdout must bind the current project id, host-validation id, provider-runner id, and sandbox-launch id, so generic reusable success output fails closed. Public manifests record only self-test exit/status and stdout/stderr/transcript hashes plus args-prefix hash/count, never helper paths, raw prefix args, or stdout/stderr text. Route callers cannot submit helper-host-ready booleans, command overrides, argv, env, binary hashes, self-test hashes, platform strings, host capability success metadata, or version strings as validation evidence. Pi consumers for this route are host-confirmed thin clients and may forward only sanitized `platform` / `notes` diagnostics, not caller helper readiness, helper hash, command, argv, env, proof-authority, or GA fields. Only a service-owned `provider_helper_host_validator` can mark the helper host ready, and even a ready host-validation manifest is still a pre-execution readiness artifact: it does not run the adapter under OS enforcement, does not collect OS-enforcement evidence, cannot satisfy the readiness gate, and must keep `adapter_execution_isolation.os_enforced=false`, `proof_authority="none"`, and `can_certify_ga=false`.

Provider-helper execution attempt manifests bind a ready provider-runner manifest to a prior append-only provider-helper host-validation artifact before any helper process can be configured or spawned. The referenced host-validation manifest must be service-owned, validated, and exactly bound to the same project, adapter, backend, provider, launch artifact, runner artifact, runner binary hash, and helper binary hash; missing, unvalidated, or mismatched host-validation artifacts fail closed before the helper config resolver runs. When binding passes, `comathd` launches the service-owned helper with `shell=false`, an optional service-owned configured or bundled args prefix, fixed argv, a non-inherited fixed `COMATH_*` environment, and a runner-binary hash match; the manifest records the host-validation artifact hash plus only executable hash, args-prefix hash/count, exit status, signal, stdout/stderr/transcript hashes, output sizes, runtime-attestation source/bound status/hash, fixed argv template hash, environment-policy hash, diagnostics, and audit metadata. It never records the helper executable path, prefix argument values, or raw stdout/stderr text. Runtime attestation is accepted only from helper stdout JSON that binds the current project id, helper-execution id, provider-runner id, sandbox-launch id, adapter, backend, provider, disabled network policy, and `proof_authority="none"`; generic reusable runtime success output remains untrusted. Caller-supplied command overrides, argv, env, exit codes, stdout/stderr hashes, host-ready booleans, runtime-attestation hashes, and success-shaped route payloads remain untrusted. A host-bound helper exit code of 0 or runtime-attestation hash proves only that the configured or bundled helper process was attempted under the service-owned protocol contract; it is not collected OS-enforcement evidence, cannot satisfy the readiness gate, and must keep `adapter_execution_isolation.os_enforced=false`, `proof_authority="none"`, and `can_certify_ga=false`.

Provider-helper collection bridge manifests bind a successful provider-helper execution attempt back to its ready provider-runner, sandbox-launch, provider-helper host-validation, and provider host-capability artifacts. That host-capability/host-validation binding is an audit and provenance prerequisite only; it is not readiness evidence by itself, not real-Pi evidence, not mathematical proof authority, and not GA certification. Route callers cannot submit OS-enforcement booleans, exit codes, stdout/stderr/transcript hashes, host-capability success metadata, provider-tool witness metadata, provider-family OS-enforcement witness metadata, provider-family execution profile metadata, provider-specific live probe attempt/execution metadata, or runtime-attestation material as collected evidence. Collection must fail closed before invoking the canonical probe writer unless the helper execution carries the current runtime-attestation binding described above, the referenced host-validation manifest still binds a matching observed service-owned provider host-capability probe, and the current provider host-capability artifact still matches the host-validation binding. Stale host-validation bindings or host-capability binding mismatch cases produce replayable blockers before configured probe output can become nested canonical evidence. When no configured or injected collection probe is supplied, the production route runs the bundled CoMath provider-helper collection probe asset with fixed service-owned argv/env. Its stdout JSON must bind the current project, collection, helper execution, runner, launch, adapter, backend, provider, helper exit code, helper stdout/stderr/transcript hashes, host-validation id/path/hash, provider host-capability probe id/path/hash, and observed host-capability status while recording false OS-enforcement facts, producing `blocked_provider_helper_collection_incomplete_os_enforcement` blocker evidence rather than readiness evidence. Operators may configure provider-specific service-owned collection probe executables through `COMATH_AGENT_ADAPTER_OSISO_OCI_COLLECTION_PROBE`, `COMATH_AGENT_ADAPTER_OSISO_NIX_COLLECTION_PROBE`, `COMATH_AGENT_ADAPTER_OSISO_FIREJAIL_COLLECTION_PROBE`, `COMATH_AGENT_ADAPTER_OSISO_WINDOWS_APPCONTAINER_COLLECTION_PROBE`, or `COMATH_AGENT_ADAPTER_OSISO_MACOS_SANDBOX_EXEC_COLLECTION_PROBE`, with bounded `*_COLLECTION_PROBE_ARGS_JSON` prefixes and `COMATH_AGENT_ADAPTER_OSISO_PROVIDER_HELPER_COLLECTION_PROBE` fallback handles. The default route executes that configured probe with `shell=false`, fixed service-owned argv/env, and disabled-network proof-authority metadata; stdout JSON must bind the same collection facts before any OS-enforcement facts are accepted. Complete provider-helper collection facts must also include a `provider_tool_execution_witness` that binds the service-owned collection probe executable hash, service-derived profile hash, fixed argv template hash, current helper transcript hash, provider, collection, helper execution, runner, launch, disabled network policy, and `proof_authority="none"`, plus provider-specific tool binding to the host-capability provider tool name/hash observed by the current service-owned capability probe, plus `provider_family_os_enforcement_witness` material bound to the current provider family, host-capability artifact/tool binding, helper transcript, complete OS-enforcement facts, disabled network policy, and `proof_authority="none"`, plus `provider_family_execution_profile` material bound to the service-derived provider-family execution kind, profile hash, and argv hash, plus `provider_specific_live_probe_attempt` material bound to that same provider-family execution profile and complete OS facts, plus `provider_specific_live_probe_execution` material from a separately configured service-owned live OS probe subprocess bound to live probe tool/argv/stdout/stderr/transcript hashes, before readiness can accept the nested canonical evidence. Readiness rechecks the current host-capability artifact path/hash, provider tool name/hash, provider-family witness hash, provider-family execution profile hash, provider-specific live probe attempt hash, and provider-specific live probe execution hash against the saved collection/evidence/probe records. Invalid binding, process failure, missing host-capability binding, missing provider-tool witness binding, missing provider-specific tool binding, missing provider-family OS-enforcement witness binding, missing provider-family execution profile binding, missing provider-specific live probe attempt binding, missing provider-specific live probe execution binding, missing facts, false facts, or stale pre-Task203/pre-Task204/pre-Task205/pre-Task206/pre-Task207/pre-Task208 evidence remain replayable blockers. A provider-specific internal service-owned collection probe can produce collected evidence only when its exit/stdout/stderr/transcript hashes exactly match the provider-helper execution manifest, it records service-owned `collection_source="service_owned_os_probe"`, helper exit `0`, all process, filesystem, network, no-new-privileges, and escape-prevention facts as true, and carries the provider-tool witness, host-capability provider tool binding, provider-family OS-enforcement witness, provider-family execution profile, provider-specific live probe attempt, and provider-specific live probe execution bindings. Incomplete facts are recorded under `blocked_provider_helper_collection_incomplete_os_enforcement`; complete facts without a witness are recorded under `blocked_provider_helper_collection_provider_tool_witness_missing`; complete facts without provider-specific tool binding are recorded under `blocked_provider_helper_collection_provider_specific_tool_binding_missing`; complete facts without provider-family OS-enforcement witness binding are recorded under `blocked_provider_helper_collection_provider_family_os_enforcement_witness_missing`; complete facts without provider-family execution profile binding are recorded under `blocked_provider_helper_collection_provider_family_execution_profile_missing`; complete facts without provider-specific live probe attempt binding are recorded under `blocked_provider_helper_collection_provider_specific_live_probe_attempt_missing`; complete facts without provider-specific live probe execution binding are recorded under `blocked_provider_helper_collection_provider_specific_live_probe_execution_missing`. The wrapper manifest is not readiness evidence by itself and must keep its own `adapter_execution_isolation.os_enforced=false`; only the nested canonical Task170 probe/evidence artifact can satisfy the readiness review, and the result still keeps `proof_authority="none"`, `can_promote_claim=false`, and `can_certify_ga=false`.

Sandbox execution probe manifests bridge that preflight to canonical OS-isolation evidence. They may bind a ready sandbox-launch manifest to a service-owned execution probe and then emit the same collected probe/evidence artifacts consumed by the readiness gate. The Pi `/cm:release agent-adapter-os-isolation-sandbox-execution` consumer is host-confirmed release tooling for this route only; it strips confirmation ids, sanitizes host paths/secrets/proof-success and transport overclaims, and cannot write `.comath/` directly. Caller-supplied request fields, Pi payloads, operator attestations, contract-only metadata, package launch metadata, and health-probe text cannot satisfy the OS-isolation readiness gate. ## Literature And Ingestion Adapters

Providers may include arXiv, Semantic Scholar, OpenAlex, Crossref, Unpaywall, Jina Reader/Search, AnySearch, local PDF, local TeX, local Markdown, BibTeX, and Zotero-compatible metadata.

Every literature evidence record must include:

- provider and source reference
- URL, DOI, arXiv ID, local artifact, or bibliography key where available
- retrieval timestamp
- content hash
- page, section, paragraph, line, or byte anchors
- prompt-injection scan result
- license/terms note
- extracted-claim kind
- `proof_authority: "none"`

Only extracted definitions, theorem statements, lemma statements, and proof-step text may become formalization candidates. They still require FormalSpecLock and Lean replay.

## Computation Adapters

Providers may include SymPy, SageMath, Python exact arithmetic, Z3, cvc5, SAT solvers, MATLAB/Mathematica/Maple when explicitly configured, and domain-specific exact scripts.

Computation reports may support:

- counterexample search
- exact computation
- symbolic exploration
- finite model search
- candidate lemma discovery
- sanity checks

They cannot promote `formally_checked` unless their certificate is formalized and accepted by Lean final replay.
